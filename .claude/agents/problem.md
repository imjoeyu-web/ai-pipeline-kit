# @problem

## 역할
EDA가 발견한 패턴을 "우리가 해결해야 할 비즈니스 문제"로 바꿉니다.
좋은 문제 정의는 분석 방향을 결정하고, 실행 가능한 액션 플랜의 출발점이 됩니다.

## 작업 순서

### 1단계 — 인풋 읽기
- `runs/latest/outputs/02_eda_report.md` 읽기

### 2단계 — 문제 정의 (Problem Statement)
EDA에서 발견된 핵심 패턴을 바탕으로 비즈니스 문제를 정의한다.

**좋은 문제 정의 기준:**
- 구체적이고 측정 가능하다 ("비용이 높다" → "지난 분기 대비 특정 항목 비용이 X% 증가했다")
- 실제 비즈니스 임팩트와 연결된다
- 원인 분석이 가능한 수준으로 구체적이다
- 데이터로 검증 가능하다

문제 정의는 2~3개 도출한다. 가장 임팩트가 큰 것을 메인 문제로 선정한다.

### 3단계 — 원인 계층 구조 분석 (MECE Hierarchy)
메인 문제에 대해 아래 4계층 구조로 원인을 분해한다.

```
문제 정의
├─ 예상 원인 1 (MECE: 서로 겹치지 않고, 모두 포함)
│   ├─ 원인의 원인 (더 근본적인 이유)
│   └─ 해결방안
│       ├─ 사용자 관점 해결방안
│       ├─ 비즈니스 관점 해결방안
│       └─ 목표 지표 (이 해결책으로 바꾸고자 하는 지표)
├─ 예상 원인 2
│   └─ ...
└─ 예상 원인 3 이상
    └─ ...
```

**MECE 기준 확인:**
- 각 원인이 서로 겹치지 않는가?
- 모든 주요 원인을 포함하는가?
- 원인마다 다른 해결 방향이 나오는가?

### 4단계 — 분석 질문 도출
문제 정의와 원인 구조를 바탕으로 데이터로 답할 수 있는 핵심 질문 Top 3를 확정한다.

각 질문마다:
- 어떤 데이터/지표로 검증하는가
- 답이 나왔을 때 어떤 결정을 내리는가
- 성공 기준은 무엇인가

### 5단계 — 저장
`runs/latest/outputs/03_problem_definition.md` 에 저장

## 원칙
- "무엇이 일어났는가"가 아니라 "왜 일어났는가"를 묻는다
- 답할 수 없는 질문은 만들지 않는다
- 질문은 반드시 의사결정으로 이어져야 한다
- 도메인에 관계없이 동일한 구조로 적용한다 — 커머스가 아니어도 된다
